1. What is GNU/ldapcppei API:

1.1 OpenLDAP

OpenLdap is a free implementation of the LDAP protocol (Lightweight Directory Access Protocol). This protocol allows the read and the update of data from directory services. OpenLdap implemente the last version of the protocol: LDAPv3. This free implementation is developed and distributed under OpenLDAP Public Licence.

OpenLdap consists of 4 parts:
  • the slapd daemon that receives and processes query requests and update the directory,
  • the slurpd daemon, replication management tool, that allows the replication and / or distribution of the directory database on various other databases, providing greater security of information stored and / or optimization of response time to client requests. Another more recent replication tool can now do this with significantly higher performance within the server daemon (slapd) by using an additional software layer (see: replication),
  • libraries offering programming interfaces, including the C language,
  • tools and utilities to manipulate the data directory, for example from commands "Shell" on GNULinux (see: ldapsearch, ldapadd, ldapmodify, ...).
The following project, GNU/Ldapcppei, is based on the C API of the OpenLdap project and provide a C++ programming interface to write C++ applications using an LDAP database (see: the GNULdapEasyAdmin project...).

1.2 The GNU/ldapcppei library

GNU/ldapcppei is a C++ library under licence GNU version 3 whose topic is to provide a C++ programming interface for easy and safe use of LDAP databases.
This library depends on the package OpenLDAP (client part only - libldap2-dev), and therefore requires its installation prior to use.

1.3 Why GNU/ldapcppei ?

As mentioned above, the purpose of this study is to provide a simple and safe interface intended for programmers of C++ code under the GNU licence:
  • easy to use for implementing the principles of object-oriented programming (well known environment from the C++ programmer),
  • standard tools for processing query results,
  • safe use, because not conducive to programming errors: no memory management to worry, eliminating the need to learn a large number of C functions specific to the LDAP protocol, the knowledge of their respective syntaxes and rules governing their sequence ...
The objective of this API is to simplify the integration of LDAP directory services in C++ sources, this reducing the number of functions (in this case, it should rather speak of "methods") needed for writing, allowing the programmer to remained focused on the management rules which he is responsible for implementation.

1.4 Advantages of the GNU/ldapcppei API ?

Besides the support for the LDAP query protocol, this interface includes some additional tools that facilitate all programming and therefore reduces the resulting source code:
  • input and output buffers to store the list of "LDAP entries", either for their next treatment, either by receiving the results of the last request; these buffers are also containers that maintain classified entries by "dn" (see: "std::set"),
  • Intermediate buffer that allows the programmer to store, for a while, the intermediate results for their reuse in subsequent treatments, possibly recursively,
  • possibility of swapping two buffers I / O in order to, for example, re-submit directly a new treatment to the outcome of its own processing,
  • operators for all the managed objects allowing an intuitive and more concise programming,
  • ...

1.5 Principles of the GNU/ldapcppei API ?

Six classes are defined:
  1. The "byteArea" class that provides a (variable) space for storing data and methods for the treatment of its contents (eg: assign, append, insert, replace, erase, fill, compare, stringEvaluate, (r)indexOf, startsWith, contains, ends, trimmed, toLower/Upper, to/fromUtf8, operateurs "+", "-=", "<", "[]", "<<", ... and many others...),
  2. Class "berValue" derived from the class "byteArea"; it instantiates the basic object: "directory data", ie a sequence of bytes of known length (note: like a "BerValue" from OpenLDAP but not to be confused with the C structure "BerValue" itself),
  3. class "attribute" that allows to instantiate the object: "directory attribute", which is composed of an identification name associated with one or more "berValue" (some directory attributes can indeed be multivalued as, for example: "memberUid"),
  4. Class "entry" allowing to instantiate the object "entry" which consists of a "dn" (Distinguish Name) and all its attached attributes,
  5. Class "ldapSet" which allows to instantiate a list of entries sorted by "dn" (see: "std::set"),
  6. and, finally, the class "ldapEasyInterface" container for directory entries (input/output) and parameters of the connection to its LDAP database.
The basic principle of this API is to define (in the class "ldapEasyInterface") an input buffer "(ldapset*)getInputSet()" that will receive the settings (ie the list of directory entries) of the LDAP queries as well as an output buffer "(ldapset*)getOutputSet()" that will receive, after query treatment, all the LDAP results (also in a resulting entries list format)...

  Operators:

    entry e1 = ... , e2 = ... , e3;
    attribute attr1 = ... , attr2=...;

    e1 += e2;       // if ( e1.dn().compare( e2.dn() ) == 0 ), or simply: if ( e1.isDn( e2.dn() ) ), all the attributes (and/or values) of "e2" will be added to "e1" or, if already present, will replace those of "e1". Equivalent to: "e1 = e1 + e2".
    e1 += &e2;    // exactly the same résult as before (ie "e1 += e2"), as for all the following examples, ie: pointers on second operands are allowed...
(Note:   in case of  "entry *e1, *e2;"  then the syntaxe should be:  "*e1 += e2;" or "*e1 += *e2"- otherwise, it would be a simple assignment of the pointer itself, of course!...).
    e1 -= e2;        // if ( e1.dn() == e2.dn() ), then all the attribute values of "e2" present in the same name attributes of "e1" will be erased from "e1". The empty attributes of "e1" will be then supressed. Equivalent to: "e1 = e1 - e2", "e1 -= &e2", "e1 = e1 - e2" or "e1 = e1 - &e2"...
    e1 &= e2;      // if ( e1.dn() == e2.dn() ), then e1 will contain only its attribute values also present in e2. Equivalent to "e1 = e1 & e2".
    e1 |= e2;       // if ( e1.dn() == e2.dn() ), then all the attributes of "e2" not present in "e1" will be added to "e1", these present in "e1" will be completed. Equivalent to : "e1 = e1 | e2", ...

    e1 = e2;         // This is an unconditional copy. Equivalent to: e1 = nullptr !... ;-)

    e1++;             // The current position in the entry "e1" is incremented (next attribute - according to the LDAP order) - equivalent to e1.gotoNext(). At the end of the attribute list, the current position will be "e1.end()", (ie null attribute), like: "e1.gotoEnd()".
                                        (Note:  if "entry *e;", the syntax must be : "(*e)++;", of course!... - and, is it necessary to precise: with (e != nullptr) ).
     e1--;               // The current position in the entry "e1" is decremented by one position (previous attribute) - equivalent to e1.gotoPrevious(). By end of the list, the current position will be "e1.begin()", ie the first attribute. It's like: "e1.gotoBegin()". Of course, if: entry *e, then the syntaxe MUST be: (*e)++, with (e != nullptr) !... Obviously... ;-)

    e1["name"];               // The current position of the entry "e1" will be positioned on the attribute of name "name" - if existing, if not: "e1.end()" ...
    e2 += e1["name"];    //  The attribute of name "name", if it exits in "e1", will be added to "e2".

Possible example:
    e3 += entry(e1 | e2)["name"];   // The attribute "name" eventually contained in "e1" or "e2" will be added to "e3". "e1" and "e2" remain themselves unchanged...

Between different object types:
    e1 += attr1;   // The attribute "attr1" will be added (except if null) to the list of attributes of "e1" or, if it is already existing, their values will complete it. Equivalent to: "e1 = e1 + attr1".
    e1 -= attr1;    // The attribute "attr1" will be removed from the attribute of the same name in "e1" and it will be deleted if, consecutively, it is empty...
    e1 &= attr1;  // The attribute "*(e1->getAttribute())" of name "attr1.name()" will be completed with the list of values contained in "attr1", only if it was already existing; equivalent to : "if ( e1.find ( attr1.name() ) *(e1.getAttribute()) += attr1;" ...
    e1 |= attr1;   // The attribute of name "attr1.name()" will be added (with its values) to the entry "e1", but only if it was not already present; equivalent to: "if ( ! e1.find( attr1.name() ) e1 += attr1;"...

  BUFFERS:
The buffers allow to save a context (list of LDAP entries) queries or intermediate results, by default: to a stack (Last In First Out) or, individually: to named buffers.

Methods of the class "ldapEasyInterface":

        - doSaveInEntries(),
        - doSaveInEntries("bufferName"),
        - doSaveOutEntries(),
        - doSaveOutEntries("bufferName").

And, to restore to the input or output set of the "ldapEasyInterface":

        - doRestoreInEntries(),
         - doRestoreInEntries("bufferName"),
         - doRestoreOutEntries(),
         - doRestoreOutEntries("bufferName").

Note: the current positions of objects are preserved (current entry, current attribute and current value...) during these operations.

1.9 The ldapEasyAdmin project?

It is a set of GUIs (frontends) to manage the various parameters of an intranet, using an LDAP database.

These GUIs are related to configuration tools that can generate the configuration files of daemon and subsystems using data of the LDAP database (especially when they are not directly interfaced with LDAP).

Example of this subsystems:
  • rsyncd (not directly interfaced with LDAP)
  • samba version 3 (partially interfaced LDAP - user identification, not on the definition of shared resources),
  • bind (interfaced with a patch, but involving severe restrictions of features),
  • users and groups, mail and proxy (almost completely interfaced LDAP - across implementations),
  • the LDAP directory server itself (début d'interfaçage ldap de la configuration d'OpenLDAP),
  • The definition of roles of specific management applications (interfaced sometimes, sometimes not...), such as for example the interfaces of this project itself (ldapEasyAdmin).
  • ...
And all that the imagination of an administrator of the intranet will be able to design...
