4.1 Overview

This GUI handles the display aspect and navigation in the LDAP tree that defines the intranet parameters stored in the corporate directory.

All GUIs - described in the following pages - of this suite  synchronize themselves according the current position in the LDAP tree represented here. As soon as the operator changes the current position within the GUI of a project, the others are automatically synchronized according the new position retained (including, for example when changing domain).

However, the actual current position does not generate necessarily an effective display of information in each of the active interfaces (depending on existing data at this new position).

While all "GNU/LdapEasyInterface" GUIs have (or will have eventually in future) some navigation/search features (sometimes limited to the current domain), in fact, the control interface shown above is the most suitable for direct selection of a new current position in directory tree.

Settings structure:

In fact, this control interface does not have specific setting. It actually uses the setting of all other interfaces "LdapEasyInterfaces" (whose it takes into account the display) to list in a tree all the managed entities (servers, persons, groups, roles, services, ...) associated with each graphic interfaces.

Of course, all these settings follow a common logic for organizing each of the entities represented in the tree display of the form: "controlFrontend"...

This "common logic" appealed to a notion of hierarchy in defining the parameters of each interface:

    ou=application
       -> ou=ldap
                -> ou=ldapEasyInterface
                         -> ou=controlFrontend
                         -> ou=domainFrontend
                         -> ou=personFrontend
                               seeAlso=ou=people
                               l=cn
                          ...
                         -> ou=serverFrontend
                                  seeAlso=ou=server,ou=computer         /* allows to define the storage RDN entity "server" in the LDAP directory */
                                  l=ou                                                            /* allows to design the storage attribute for the entity "server" (optional) */
                                  -> ou=sambaConfigFrontend
                                           seeAlso=ou=sambaConfig
                                           l=ou
                                           -> ou=sambaShareFrontend
                                                    seeAlso=ou=shareDefinition
                                                    l=sambaShareName
                                  ->ou=rsyncConfigFrontend
                                        ...
                                  ->ou=nfsConfigFrontend
                                        ...
                                  -> etc.
Thus, we can see above that the interface settings "serverFrontend" - managing the "server" entity - hosts in its tree an other definition of parameters, here the settings of the interface; "sambaConfigFrontend" - managing the "samba service" entity - which, itself, in turn hosts a third parameter definition, that of the interface "sambaShareName" - managing the "samba ressources" - ...

Why this nesting?
This allows to indicate to the display interface "controlFrotend" that entities "samba services" eventually defined in the database belong each to a "server" and will be displayed, if existing, in each of their parent server.

In other words, for each server to display, it must be ensured whether the notion of "service samba" is defined, and if so, display it as a subfolder "sambaConfig". This mechanism is recursive (see: "sambaShare").

It may be the true for any other service defined and possibly managed by an existing interface "LdapEasyInterface", such as for example "rsyncFrontend", "nfsFrontend", ...

The organization of the LDAP data structure associated with this parameterization is therefore as follows:

    dc=mydomain
       -> ou=server,ou=computer
          -> ou=11:22:33:44:55:66
                      description=...
                      ...
                      -> ou=sambaConfig
                              -> ou='netbiosName'
                                      -> sambaOptionName=description
                                      -> sambaOptionName=...
                                      ...
                                      -> ou=shareDefinition
                                              -> sambaShareName=`shareName1`
                                                      -> sambaOptionName=description
                                                      -> sambaOptionName=...
                                                      ...
                                              -> sambaShareName=`shareName2`
                                                      -> sambaOptionName=description
                                                      -> sambaOptionName=...
                                                       ...
